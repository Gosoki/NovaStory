## 1. 研究核心观点(不变,论文灵魂句)

> **好的 AI 共创,应该在生成之前就把问题规避掉,而不是生成之后让用户反复发现问题、要求修改。**
> 小白有很多细节(主角心理、转折、画面重点)不会写进提示词,AI 只能按默认值补齐;像 AskUserQuestion 一样**先问清楚再动手**,产出更贴合心中所想——更高效、更少返工。

三条件 = 人类意图进入流水线的时机:

| | C 一次成型 | D 先生成再修改 | E 先引导再生成 |
|---|---|---|---|
| 生成前 | 创意 1-2 句 | 创意 1-2 句 | 创意 + **结构化引导问答** |
| 生成后 | 只能提交 | 修改循环(自由反馈 + 手改) | 修改循环(继续引导 + 手改) |
| 对应现实 | 一键生成工具 | **常规 ChatGPT 工作流**(生成→喂反馈) | 本研究提出的工作流 |

核心假设:E 的事前投入能**大幅减少事后修改**(轮数/手改量/时间),并带来更高的意图保真与所有权。D vs C 检验"光有事后修改权值多少"。

## 2. 三条件最终交互规格(Q1-Q6 已定)

### C · 一次成型(无任何回路)
```
创意 → AI 生成完整分镜稿 → 看稿 → [提交]
```
不能改、不能反馈(零人类回路锚点)。

### D · 先生成,再修改(双通道修改循环)
```
创意 → AI 生成完整分镜稿 → 看稿 →
   [提交]
   [告诉 AI 怎么改]   ← 自由文字框:可抽象("更滑稽""更炸裂")也可具体
   [自己改稿]         ← 直接在稿子文本框上手编辑
   → AI 修订(基于用户最新稿)→ 回到看稿,无上限循环
```
**页面文案须明示**三种玩法:可以给 AI 抽象方向、可以提具体细改、也可以自己动手改。

### E · 先引导,再生成(引导问答 + 修改循环)
```
创意
→ 【第 1 轮引导】AI 一次生成 5-7 问:
     固定 3 维(主角心理 / 关键转折 / 核心画面)
   + AI 根据用户创意补充 2-4 个其他维度的问题(帮用户抵达心中理想)
     每问 = 问题 + 3-4 选项 + 「自己写」 + 「AI 帮我决定」
     逐条展示,答完一并提交
→ AI 生成完整分镜稿(**无大纲层,Q1 已砍**)→ 看稿 →
   [提交]
   [让 AI 继续引导]   ← AI 读当前稿(含用户手改),出 1-3 个新问题 → 作答 → AI 修订
   [自己改稿]         ← 同 D 的手改
   → 回到看稿,无上限循环
```
E **没有**自由文字反馈通道(那是 D 的机制;用户想表达具体想法可走"自己写"或手改)。

### 看稿循环的交互细节(v4.1,按用户 2026-06-13 批示修订)

- **看稿页(D/E)** = **脚本本身就是常驻的可编辑文本框**(用户随时直接改,无需点任何按钮)+ 条件专属 AI 通道**常驻**(D:「告诉 AI 怎么改」聊天输入框+发送按钮;E:「让 AI 继续引导」按钮)+ [提交] 按钮。
- 手改的持久化:用户改完不需要手动保存——**点提交或任何 AI 动作时自动对比文本框与当前版本,有差异先落一版 `author=user_edit`**(手改次数/改动字符量由此记录)。C 的看稿页仍为只读渲染+提交(零回路)。
- **E 的引导问答 UI**(首轮与追加轮同一组件):逐条展示"第 i/N 问",每问 = 问题 + 一行 why 小字 + 选项按钮 + 「✍ 自己写」 + 「🤖 AI 帮我决定」;可回看上一问;全部作答后一并提交。
- **等待文案**:引导问题生成期间显示"AI 正在为你准备问题,约 1 分钟…"(C2 实测延迟较高,见 §4)。
- **降级**:引导 JSON 解析重试后仍失败 → 该轮退化为一个开放问题("用几句话补充你想要的方向"),记 `fallback=true`,流程不中断。

### 全条件通用硬规则(用户特别强调)

> **任何"提交"或"交给 AI"(修订/引导)动作之前,必须先把用户当前手改后的稿子持久化为最新版本**;AI 的修订永远基于用户最新稿。每一版稿子快照入库并标注作者(`ai` / `user_edit`)。实现位:三按钮的处理函数第一步统一调用"落盘未保存手改"。

## 3. Q4 可行性分析(E 允许手改,记录在案)

用户决定:E 看稿后给三个选择(提交 / 继续引导 / 手动修改),理由是"和 D 对比依旧是引导式,前期选择能大幅减少后期修改"。可行性结论:**成立**,理由与代价:

- ✅ **假设反而更可检验了**:E 与 D 共享"手改+提交",唯一系统性差异 = 事前引导 + AI 通道类型。核心主张"事前引导减少事后修改"现在能被**直接测量**——比较 D/E 的事后修改总量(AI 轮数 + 手改字符量 + 事后耗时)。E 用户若仍大量手改,假设被证伪——可证伪性是优点。
- ✅ 生态效度高:不许手改才是反常的;且"E 用户其实不怎么用引导、光手改"这种行为本身就是有价值的发现。
- ⚠️ 代价(已接受):D vs E 是双差异(事前引导 + AI 通道型),不是单变量——路 B 系统评估框架下成立,论文里把 D 明确定位成"**常规 ChatGPT 工作流**"的代理,叙事就顺了:"提出的工作流 vs 现状工作流"。
- ⚠️ 工程上必须做对:手改快照规则(§2 末)+ 手改量逐版记录,否则"事后修改量"测不出来。

## 4. 结构化返回与模型策略(v4.1:C2 已测,结果回填)

- **C2 测试结果(2026-06-13)**:mimo-v2.5-pro 质量 **8/8 严格通过**——8 组创意(含"想拍一个搞笑的"这种极端模糊输入)全部产出合规 JSON:3 固定维齐全、补充维 2-3 个且贴题(tone/ending/sound)、每问 4 个具体互斥选项、问题紧扣创意;追加轮冒烟亦通过(3 问且准确点中稿子真实薄弱处)。**质量门槛通过。**
- **遗留:延迟 62-84s/调用(均值 ~75s)**。处置:开发期用 mimo;引导步模型做成可独立配置(`GUIDANCE_API_INDEX`,默认跟随主配置);**正式收数前重测,仍 >30s 则引导步指向 OpenAI**(gpt-4o-mini 单次 ≈$0.001-0.003,30 人全程 <$1)。
- 健壮性不变:严格 JSON + 重试 2 次(重试时回喂解析错误)+ 降级开放问题记 `fallback=true`;尽量避免纯开放。C2 的测试 prompt 即 C1 的 v0 草稿。

## 5. 意图基准与测量(不变)

创意 1-2 句 = 客观文字基准;主观侧靠"想象匹配度"题;辅以逐镜头标注、违背本意题。

## 6. 数据记录规格(实现依据)

```json
trials 增改:
  guidance_json: {
    "rounds": [
      {"round": 1, "source": "fixed3+ai_supplement", "items": [
        {"dimension": "...", "question": "...", "options": [...],
         "chosen": "...", "is_custom": false, "ai_decided": false, "fallback": false}
      ]},
      {"round": 2, "source": "ai_from_draft", "draft_snapshot_ref": "v3", "items": [...]}
    ]
  },
  revision_requests: [ {"round": 1, "text": "更滑稽一点"} ],     // 仅 D
  script_versions: [ {"v": 1, "author": "ai"}, {"v": 2, "author": "user_edit"}, ... ],
  n_ai_rounds, n_hand_edits, hand_edit_chars                      // 行为汇总列
```

核心行为指标:**事后修改总量**(AI 轮数 + 手改量 + 事后净时间)、事前投入(E 第 1 轮作答、自填率、AI帮我决定率)。token 用量后期加(paper/8 C6)。
